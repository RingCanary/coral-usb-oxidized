## Result

### Richer-divisor same-product scans
Across all four new p64 same-product scans, each product still contains exactly one `EO=6496` member:

- `896`: `8x112`
- `960`: `8x120`
- `1152`: `8x144`
- `1280`: `8x160`

The remaining members in each product split into distinct non-`6496` families rather than forming a second product-local `6496` partner. So the richer-divisor pass did **not** produce a promotable same-product `6496` family.

### Explicit mixed-product local neighborhood
The explicit probe revealed a stronger structure than the earlier “singleton island” reading:

- `k3_eo6496_pc688_param37376`: `8x112`, `8x120`, `8x128`, `8x136`, `8x144`
- `k3_eo6512_pc688_param37376`: `12x88`, `12x96`, `12x112`, `16x72`, `16x80`
- `k3_eo8176_pc688_param37376`: `7x128`, `7x144`
- `k3_eo8464_pc688_param37376`: `9x112`, `9x128`
- `k3_eo8656_pc688_param37376`: `10x96`, `10x112`, `10x128`, `14x80`, `14x96`

This is the decisive refinement:

> `EO=6496` is a local `H=8` band across nearby products, not just the single point `8x128`.

But the promotion gate still fails, because each `6496` member belongs to a different product:

- `896`: `8x112`
- `960`: `8x120`
- `1024`: `8x128`
- `1088`: `8x136`
- `1152`: `8x144`

So the current family schema still has no nontrivial same-product `6496` family to freeze.
